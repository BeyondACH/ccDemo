## 2）Code Review 模板：`.ai/templates/code-review.md`

```md
# Code Review

> PR：
> Reviewer：
> 关联设计/Spec：
> 时间：YYYY-MM-DD
> 风险等级：低 / 中 / 高

## 1. 总体评价
- 做了什么：
- 是否符合需求/设计：
- 总体质量印象：

## 2. Blocking Issues（必须修）
> 这些问题不修不建议合并。每条都要可执行、可定位。

1) **[Correctness]** <一句话描述>
- 位置：`path/to/file:line`（如果能定位）
- 影响：会导致什么错误/风险
- 复现/证据：日志/测试/推理
- 建议修复：最小改动方案

2) ...

## 3. Non-blocking Suggestions（建议改）
> 这些不阻塞合并，但能提升质量/可维护性。

- **[Maintainability]** ...
- **[Readability]** ...
- **[Performance]** ...
- **[DX]** ...

## 4. 设计一致性检查（Spec Alignment）
- [ ] 接口/行为与 spec 一致
- [ ] 边界条件覆盖（空值/越界/并发/重试/幂等）
- [ ] 兼容性（老客户端/老数据）
- [ ] 数据迁移/回滚策略有落实（如适用）

## 5. 正确性（Correctness）清单
- [ ] 状态机/业务规则正确
- [ ] 错误处理与返回码一致
- [ ] 幂等/重试安全
- [ ] 时间/时区处理正确
- [ ] 并发与竞态条件处理（锁/事务/原子性）

## 6. 安全（Security）清单
- [ ] 鉴权/鉴权绕过风险
- [ ] 输入校验（注入/越权/反序列化等）
- [ ] 敏感信息泄露（日志/响应/异常）
- [ ] 文件/路径/命令执行风险（如适用）
- [ ] 依赖升级风险（CVE/许可）

## 7. 性能（Performance）清单
- [ ] 热路径无明显 O(n^2) 或 N+1 查询
- [ ] 数据库索引/查询合理（如适用）
- [ ] 大对象/大列表有分页/限制
- [ ] 缓存策略合理（如适用）
- [ ] 资源释放正确（连接/线程/文件句柄）

## 8. 可维护性（Maintainability）清单
- [ ] 模块边界清晰、职责单一
- [ ] 命名清晰、注释必要且准确
- [ ] 重复代码可接受或已抽取
- [ ] 配置可控（常量/开关/环境变量）
- [ ] 日志/指标/追踪（可观测性）到位

## 9. 测试（Testing）清单
- [ ] 单元测试覆盖关键逻辑
- [ ] 集成测试覆盖关键链路（如适用）
- [ ] 异常分支/边界值测试
- [ ] 测试稳定、可重复、无随机失败
- [ ] 提供本地/CI 运行命令

## 10. 文档与发布（Docs & Release）
- [ ] README/变更说明更新（如需要）
- [ ] 配置项/环境变量说明
- [ ] 数据库迁移脚本与回滚
- [ ] 监控/告警更新（如需要）

## 11. 建议的合并前动作（Action Items）
- [ ] 修复所有 Blocking
- [ ] 补测试：
- [ ] 补文档：
- [ ] 再跑一遍：
  - `...`（lint/test/build）
